# 交互工作台与结构化表达 — 目标

本文只写**意图与约束**，**实现方式**（存储、协议、具体布局、工具接入等）留给后续设计。

---

## 目的

本仓库覆盖对话、语音、虚拟形象、游戏与大量集成。本方向是**增量**的：让人与角色能在**结构化表面**上共同使用**共享产物与输出**，而不只是一份线性记录。**聊天气泡**继续作为合法、主要的呈现与交互载体之一（含富文本与工具结果等）。

**成功标准**侧重：复用会话、插件、工具结果等**共享能力**，避免各功能各写一套临时实现。**语音合成（TTS）、唇形同步等**不在本路线内展开，由既有音频/形象管线与其他模块演进。

## 工程约定（适用于本方向相关改动）

1. **优先通用扩展点**
   若现有代码需改动以支持工作台相关能力，倾向暴露**体量小、可文档化、可复用**的 API（钩子、注册表、协议类型、IPC 约定等），使**其他功能**（开发者工具、游戏、其他界面）在**不**依赖工作台专有类型的情况下也能使用。

2. **保持可读与可维护**
   避免一次性强耦合（「只有工作台 import 这一处」），除非范围明确为实验且已隔离（如特性开关或独立包）。

3. **沿用现有模式**
   与仓库已有做法衔接：角色 / 对话 / 会话归属、服务端通道与上下文、插件、Eventa、多界面 stage-ui 等——**具体接线按功能决定**。

4. **公开接缝要可发现**
   新引入的 API 应在归属包或模块旁有命名与说明，即便首个消费者仍是实验性质。

以上**不是**鼓励过早抽象：含义是「若改动共享模块，先想清楚接缝是工作台专用还是全项目可用」。

## 产品目标（收敛后）

### G1 — 共享产物

用户与智能体通过**稳定标识**指向**同一对象**（图示、表格、卡片、链接、结构化包等），生命周期服从会话或产品规则（持久、分叉、回放等——**语义待定**）。**聊天气泡**可承载、引用这些产物，并与结构化展示协同。

### G2 — 结构化、面向受众的输出

角色**有意呈现**的内容（看板、图、表、列表、对比等）可以**结构化数据**形式交给界面渲染；与**对话文案**在同一语境下可读、可指称。

### G3 — 注意力与节奏

产品能**引导焦点**：当前在讨论什么与背景上下文如何区分（先框架后细节、高亮、滚动或镜头语义等——**机制待定**）。目标是降低认知负荷，而非绑定某一动画库。

### G4 — 文案与可视产物一致

屏幕上的文字与可视化产物使用**同一套指称**（「这个节点」「那块板」），避免各说各话。**不**在本路线内规定 TTS、字幕或形象嘴型如何跟进（由其他模块负责）。

### G5 — 插件与模块参与（轻量）

内部（或可选外部）模块能以与现有**插件 / 工具结果呈现**一致的方式**贡献**渲染或扩展（**形态与加载方式待定**）。与 #840 中的 **MCP 连接器**、更广**协议**线有交集时，以「对话内工具块」与「其他展示面」如何分工为开放问题，本文不拍板。

## 问题陈述（缺口），非方案

- 跨对话、会话持久化与可选结构化 UI 之间，尚缺统一的 **artifact（产物）** 一等概念。
- **工具调用**与「对用户可见的输出」尚未统一在 **焦点 + 产物** 模型下描述。
- **非纯对话区域**的插件贡献方式，可能与未来的 artifact / 工作台 API 需要对齐。
